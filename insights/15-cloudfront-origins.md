## What an Origin Is
An origin is the source where your original content lives. It is the place CloudFront goes to fetch content when it receives a request for something it does not have in its cache. The origin holds the authoritative copy of every file. CloudFront does not create content or modify it. It only retrieves content from the origin, caches it, and serves it to users.

## Why the Origin Matters
The origin determines where CloudFront fetches content on a cache miss. If the origin is unavailable, misconfigured, or denying CloudFront's requests, users will receive errors even if CloudFront is working correctly. The origin is the foundation of the entire delivery chain. Everything CloudFront serves ultimately comes from the origin, which means the origin's availability, permissions, and configuration directly affect what users experience.

## How CloudFront Fetches From an Origin
When a user requests content that is not in the cache at the nearest edge location, CloudFront sends a request to the origin asking for that content. The origin processes the request and returns the content. CloudFront receives the content, stores a copy in the edge location's cache, and serves it to the user. Every subsequent request for the same content from that edge location is served from the cache without another trip to the origin.

CloudFront signs its requests to the origin using the credentials associated with the Origin Access Control or the origin's access settings. The origin checks the incoming request to verify it has the correct credentials before responding. If the credentials are missing or incorrect, the origin returns an error and CloudFront passes that error back to the user.

## S3 Origins
An S3 origin uses an S3 bucket as the source of content. When CloudFront receives a request, it fetches the corresponding object from the S3 bucket and returns it to the user. S3 origins are the most common type for static websites because S3 is reliable, inexpensive, and integrates directly with CloudFront through Origin Access Control.

When you use an S3 bucket as an origin, you have two options for how CloudFront connects to it. The first is using the S3 REST API endpoint, which is the standard way to access S3 objects programmatically. The second is using the S3 static website endpoint, which is the URL generated when you enable static website hosting on the bucket. These two endpoints behave differently in important ways.

## Custom Origins
A custom origin is any HTTP server that is not an S3 bucket. This includes EC2 instances running web server software, Application Load Balancers, on-premises servers, and third-party hosting providers. Custom origins give you the flexibility to use CloudFront in front of any web server, not just S3.

Custom origins are used when your content is dynamic, generated by server-side code, or requires backend processing that S3 cannot provide. A website that queries a database and generates pages on the fly cannot use S3 as its origin because S3 only serves static files. It needs a custom origin that runs the server-side code and returns the generated response.

## How S3 REST API Endpoint Differs From S3 Website Endpoint
The S3 REST API endpoint is the standard endpoint for accessing S3 objects. It supports Origin Access Control, which keeps your bucket private while allowing CloudFront to read from it. Requests to the REST API endpoint require authentication unless the bucket is public. This is the endpoint AWS recommends using with CloudFront because it supports OAC and HTTPS between CloudFront and S3.

The S3 static website endpoint is the URL generated when you enable static website hosting on the bucket. It does not support Origin Access Control. To use the website endpoint as a CloudFront origin, the bucket must be publicly accessible. It supports HTTP but not HTTPS for the connection between CloudFront and S3. The website endpoint is useful when you need S3 to handle index documents in subdirectories, which the REST API endpoint does not support.

In this workshop, you use the S3 REST API endpoint as your origin and configure Origin Access Control to keep the bucket private while allowing CloudFront to read from it.

## What Happens When the Origin Is Misconfigured
If the origin domain is entered incorrectly when creating the distribution, CloudFront cannot find the origin and every request results in an error. If the bucket policy has not been updated to grant CloudFront access, S3 returns an Access Denied error on every request. If the origin is an S3 website endpoint but the bucket is not public, requests return an Access Denied error because the website endpoint does not support OAC.

## Key Terms
Origin — the source where the original content is stored, from which CloudFront fetches content when it does not have a cached copy at the edge location.

S3 origin — an S3 bucket configured as the source of content for a CloudFront distribution.

Custom origin — any HTTP server that is not an S3 bucket, used as the source of content for a CloudFront distribution when dynamic content or server-side processing is required.

S3 REST API endpoint — the standard endpoint for accessing S3 objects programmatically, which supports Origin Access Control and HTTPS between CloudFront and S3.

S3 website endpoint — the URL generated when static website hosting is enabled on an S3 bucket, which does not support Origin Access Control and requires the bucket to be publicly accessible.

Cache miss — when an edge location does not have a cached copy of the requested content and must fetch it from the origin.

Origin Access Control — the permission configuration that allows CloudFront to read from a private S3 bucket using signed requests while denying direct access to everyone else.

## What OAC Is
Origin Access Control is a CloudFront feature that allows you to keep your S3 bucket completely private while still allowing CloudFront to read from it and deliver its contents to users. Without OAC, serving content from a private S3 bucket through CloudFront requires making the bucket publicly accessible, which exposes your files to anyone who discovers the bucket URL. OAC eliminates that requirement by giving CloudFront a secure, authenticated way to access the bucket without making it public.

## Why OAC Exists
Before OAC, the standard way to serve S3 content through CloudFront was to make the S3 bucket publicly readable. This worked but created a security problem. A publicly readable bucket means anyone who knows the bucket URL can access your files directly without going through CloudFront, bypassing your CDN, your caching, and any security rules you have configured. If your bucket URL became known, you would have no way to control who accessed your content or prevent direct access that avoided CloudFront entirely.

OAC solves this by creating a trust relationship between a specific CloudFront distribution and a specific S3 bucket. CloudFront signs every request it makes to S3 with a cryptographic signature. The bucket policy is configured to accept only requests that carry the correct signature from the correct distribution. Everything else is denied.

## How OAC Works
When a user requests content through CloudFront and the edge location does not have it cached, CloudFront needs to fetch it from S3. With OAC configured, CloudFront signs the request using AWS Signature Version 4 before sending it to S3. This signature includes information about the request, the distribution making the request, and a timestamp.

S3 receives the signed request and checks it against the bucket policy. The bucket policy specifies which CloudFront distribution is allowed to read from the bucket. If the signature matches and the distribution is the one named in the policy, S3 serves the requested object. If the signature is missing, incorrect, or from a different distribution, S3 returns an Access Denied error.

## The Two Steps Required
Configuring OAC requires completing two separate steps in two different services. Both steps are required. Completing only one of them results in an Access Denied error.

### Step 1: Creating the OAC in CloudFront
In the CloudFront console, you create an Origin Access Control and attach it to your distribution's origin settings. This tells CloudFront to sign every request it makes to S3. After you create the OAC, CloudFront generates a bucket policy that grants the specific permission needed. You must copy this policy before moving to Step 2.

### Step 2: Applying the Bucket Policy in S3
In the S3 console, you navigate to your bucket's permissions settings and paste the bucket policy that CloudFront generated. This tells S3 to accept signed requests from your specific CloudFront distribution and deny everything else. Until this policy is in place, S3 has no rule granting CloudFront access, and every request from CloudFront returns an Access Denied error regardless of whether the OAC is correctly configured in CloudFront.

## Why Both Steps Are Required
CloudFront and S3 are separate services with separate permission systems. Creating the OAC in CloudFront only changes how CloudFront behaves when making requests to S3. It does not automatically update S3's permissions. S3 still needs to be told explicitly which requests to accept. That instruction comes from the bucket policy you paste in Step 2. Without it, S3 treats CloudFront's requests the same way it treats any other unauthorized request and denies them.

## The Access Denied Error and What Causes It
An Access Denied error when visiting your CloudFront URL means S3 denied CloudFront's request for the file. This almost always means one of three things. The first is that Step 2 was not completed and the bucket policy has not been pasted into S3. The second is that the bucket policy was pasted incorrectly or into the wrong bucket. The third is that Block Public Access settings in S3 are overriding the bucket policy and preventing CloudFront from reading the files.

If you see an Access Denied error after completing both steps, go back to your S3 bucket's permissions tab and verify that the bucket policy is present and matches the one generated by CloudFront. Then check that Block Public Access is not overriding the policy.

## OAC vs Making the Bucket Public
Making the bucket public is simpler to configure but creates a security exposure. Anyone who discovers your S3 bucket URL can access your files directly without going through CloudFront, bypassing your CDN, your caching, and any WAF rules you have configured. Your files are exposed to the public internet with no way to control or audit who accesses them.

OAC keeps the bucket private. Only CloudFront can read from it. Users always access your content through CloudFront, which means your caching, your security rules, and your HTTPS configuration are always in effect. The bucket URL is useless to anyone who does not have the OAC signature, which only CloudFront can generate.

## Key Terms
Origin Access Control — a CloudFront feature that allows a specific CloudFront distribution to read from a private S3 bucket using signed requests while denying direct access to everyone else.

AWS Signature Version 4 — the cryptographic signing method CloudFront uses to authenticate its requests to S3 when OAC is configured.

Bucket policy — the JSON document pasted into S3 that tells it to accept signed requests from a specific CloudFront distribution and deny all other requests.

Access Denied — the error returned by S3 when a request does not have the correct permissions to access the requested resource, most commonly caused by a missing or incorrect bucket policy.

Signed request — a request that includes a cryptographic signature generated by CloudFront, which S3 uses to verify that the request is coming from the authorized distribution.

Block Public Access — S3 settings that override bucket policies to prevent public access, which can interfere with OAC if misconfigured.
